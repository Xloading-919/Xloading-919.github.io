Spring Bean 生命周期

Spring Bean 生命周期这个点之前都是背过那9步。但是具体的每个步骤是有什么意义一直没有研究过。这次来看看每个步骤有什么意义。

1. Bean定义的加载与解析#

在Spring IOC 容器启动时，会把配置文件、注解、配置类中的写的Bean信息加载到一个Map中，叫做BeanDefinition，其中存储了类名、作用域、延迟初始化等元数据。

Bean可以通过三种方式定义：

  • XML 配置文件：传统方式，通过 <beans> 根标签和嵌套的 <bean> 标签声明，支持属性注入（<property>）、构造函数注入（<constructor-arg>）等配置。
  • 注解扫描：通过 @ComponentScan 开启组件扫描，结合 @Component 及其派生注解（如 @Service）自动注册 Bean。支持 @Autowired 实现依赖注入。
  • Java 配置类：使用 @Configuration 标记配置类，通过 @Bean 方法显式定义 Bean。适用于需要编程式初始化的场景，如第三方库集成

2. 实例化Bean#

首先是把Bean进行实例化。容器找到配置文件中的Bean定义，通过反射创建实例。

然后开始实例化。实例化是在Spring启动过程中，刷新上下文的时候执行的。

  • 在 AbstractApplicationContext.refresh() 方法调用时
  • 调用 finishBeanFactoryInitialization() 方法，其中通过调用preInstantiateSingletons() 触发
  • 会调用instantiateSingleton()方法加载，内部是调用了getBean()方法
  • getBean()方法会调用doGetBean()方法，这里面会触发创建逻辑，调用createBean()方法，其中调用 doCreateBean 方法
  • 这个 doCreateBean 方法是 AbstractAutowireCapableBeanFactory 类中的核心方法，负责实际创建和初始化Bean实例。其中的 createBeanInstance 方法负责实例化Bean。

这里会先判断是否需要提前暴露单例Bean，如果需要提前暴露，会添加单例工厂到三级缓存（singletonFactories）。

3. 属性赋值#

设置对象属性，依赖注入

这一步还是在上面的 doCreateBean 方法中进行。调用populateBean()方法。这里会做实例化后处理、自动装配处理、属性后处理、依赖检查。最后把最终的属性值应用到Bean实例上。

4. 初始化#

  1. 检查是否实现了Aware相关的接口并设置依赖
  2. 执行BeanPostProcessor对象中的前置处理
  3. 是否实现InitializingBean接口
  4. 是否定义init-method方法
  5. 执行BeanPostProcessor对象中的后置处理

这里初始化的操作分为几步，在上面属性填充后，会调用 initializeBean 方法，在其中分别执行每一步。

Aware相关的接口#

调用 invokeAwareMethods 方法，检查是否实现了 Aware 标记接口。如果实现了，继续检查是否实现各种 Aware 子接口。

这个方法允许Bean在初始化过程中获取到Spring容器的相关信息。

  • Bean名称：通过 BeanNameAware.setBeanName
  • 类加载器：通过 BeanClassLoaderAware.setBeanClassLoader
  • Bean工厂：通过 BeanFactoryAware.setBeanFactory

这样Bean就可以在需要时访问容器资源或了解自己的上下文信息。可以在自己定义的Bean内部，设置Bean名称、类加载器、Bean工厂。

执行BeanPostProcessor对象中的前置处理#

调用 applyBeanPostProcessorsBeforeInitialization 方法，用于在Bean初始化之前应用所有注册的 BeanPostProcessor 处理器。

比如AOP代理创建、Bean增强、处理特定注解或配置。

  • CommonAnnotationBeanPostProcessor：处理 @PostConstruct、@PreDestroy 等 JSR-250 注解。
  • AutowiredAnnotationBeanPostProcessor：处理 @Autowired 和 @Value 注入。
  • AbstractAdvisingBeanPostProcessor：AOP 代理的基础实现类。

InitializingBean接口和init-method方法#

调用 invokeInitMethods 方法，用于调用Bean的初始化方法。在其中包括两个部分：

  1. 先检查是否实现了 isInitializingBean 接口，这个接口中有 afterPropertiesSet 方法，用于执行Bean属性设置完成后的初始化逻辑。
    1. 比如：初始化一些资源（数据库连接池）、加载配置文件
  2. 然后可以获取自定义的init-method初始化方法，并执行。
    1. 自定义初始化方法必须是无参数、无返回值，且不能抛出 checked 异常的方法
    2. 若同时实现 InitializingBean 和配置 init-method，则 afterPropertiesSet() 先执行，自定义初始化方法后执行
    3. JSR-250 规范中的 @PostConstruct 注解，其作用与 init-method 一致，且优先级高于 init-method（Spring 会先执行 @PostConstruct 标注的方法，再执行 init-method）
    4. 配置方式：
      • XML 配置：<bean id="userService" class="com.example.UserService" init-method="init"/>
      • Java 配置：@Bean(initMethod = "init")
      • 注解配置：@PostConstruct

执行BeanPostProcessor对象中的后置处理#

调用 applyBeanPostProcessorsAfterInitialization ，用于在Bean初始化之后应用所有注册的 BeanPostProcessor 处理器。

比如AOP代理创建、Bean包装和增强、Bean验证。执行结束后返回包装的Bean。

  • AOP 实现的核心：这一阶段是 Spring 实现 AOP 的核心环节 —— Spring 会在此处为需要代理的 Bean 生成代理对象（如基于 JDK 动态代理或 CGLIB 代理），并将代理对象返回给容器。后续容器中存储和使用的，就是这个代理对象。

5. 销毁#

  1. 在使用之前，注册Destruction相关回调接口
  2. 销毁阶段：根据是否实现DisposableBean接口
  3. 是否配置了destory-method进行处理

注册销毁时回调方法#

首先，还是在 doCreateBean 方法中，调用 registerDisposableBeanIfNecessary 注册Bean销毁时的回调方法。注意：如果Bean没有实现 DisposableBean 接口，也没有配置 destroy-method，则不需要销毁回调，也就不会创建回调。

如果是单例Bean：调用 registerDisposableBean 注册销毁适配器。使用 DisposableBeanAdapter 包装Bean的销毁逻辑。

这样Bean创建完成，返回Bean。然后就是进入使用阶段。

具体销毁阶段#

当IOC容器关闭时（如 Web 应用停止、调用 ApplicationContext.close() 方法），会对容器中的Bean进行销毁处理，释放资源。

在 AbstractApplicationContext 类的 close() 方法中，调用 doClose() 方法。最终调用 destroySingletons() 方法。

  1. 在 DefaultSingletonBeanRegistry 实现类里，有 destroySingletons() 对于单例Bean的实现，最终会调用上面注册适配器的 destroy() 方法。这个方法是 DisposableBean 接口定义的。在其中可以执行具体的销毁逻辑。
  2. destroy() 方法中会根据配置的自定义 destroyMethods 进行调用处理。

与初始化类似，自定义销毁方法可以通过配置指定。

  • XML 配置：<bean id="userService" class="com.example.UserService" destroy-method="cleanup"/>
  • Java 配置：@Bean(destroyMethod = “cleanup”)
  • 注解配置：@PreDestroy
    • JSR-250 规范中的 @PreDestroy 注解，其作用与 destroy-method 一致，且优先级高于 destroy-method（Spring 会先执行 @PreDestroy 标注的方法，再执行 destroy-method）

仅单例 Bean（默认作用域）会被 Spring 容器管理销毁过程
